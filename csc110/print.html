Grading

The grading system for my classes is loosely based on labor-based grading. It might be a bit different from what you’re used to, but it should be easy enough to understand:

  • Assignments are graded as follows:
    • An assignment is broken into one or more units of work, which I’ll refer to as tasks.
    • Each task has a point value and is marked either complete (you get 100% of the task’s points) or incomplete (you get 0 points for the task) when your work is graded. There is no partial credit for incomplete tasks.
    • Most assignments allow multiple submissions, and only your best submission counts.
  • Your final grade is the sum (not the average) of your assignment grades and uses the letter grade scale in QU’s grading system. This means five points on an assignment is equal to five points on your final grade.
  • There are no exams. Your final grade is only determined by your assignment grades.

Commands

You interact with a terminal by typing a command and reading the command’s output. This section will cover the structure of a command and some terms you’ll need to know to understand these commands.

There are differences between Windows and Unix-like systems, but most of these differences won’t matter if you are using a bash emulator on Windows. For example, Windows uses a backslash to separate directory names in paths, but your bash emulator should allow you to use a forward slash as if you were on a Mac or Linux computer. My notes focus on Unix-like systems, but I will try to note differences between Git Bash and bash/zsh when they come up.

Terms

Some terms that will be used to explain terminal commands. Refer back to this list if you come across something you don’t understand.

  • Directory: another word for a folder on your hard drive. A directory can contain files and other directories.
    • Your working directory is the directory in which your terminal (or another program) is currently running. Your terminal has direct access to files in its working directory.
    • Your home directory is the where the terminal starts. This should be your user folder: /home/username on Linux, /Users/username on Mac, and C:\Users\username (displayed as /c/Users/username in a bash emulator) on Windows.
  • Path: a path is the location of a file or directory. It consists of directory names separated by forward slashes (/). The path ends with the name of the file or directory it leads to.
    • An absolute path tells you how to reach a file or directory from the root of a file system. An absolute path starts with a forward slash on Unix-like systems. For example, the absolute path to my Desktop can be written as /home/zach/Desktop or /home/zach/Desktop/. The forward slash at the end of the path is optional, and it can only be included when the path points to a directory (such as the Desktop folder).
    • A relative path tells you how to get from one location (usually your working directory) to a specific file or directory. For example, if I were in the directory /home/zach and wanted to reach /home/zach/Desktop/notes.txt, then the relative path Desktop/notes.txt tells me how to reach that file from my current location.
    • Your computer also has a list of paths that it checks whenever you ask it to run a program. This list is stored under the label “path” or “PATH”, and we can just refer to it as “the path”. When you install a new program, the installer will usually add its location to the path or create a shortcut in a directory that was already on the path. If a program is on the path, then you only have to type its name to run it. If a program is not on the path, then you have to type a path to that program to run it.

Command Structure

Commands on the terminal can refer to programs or built-in functions. Sometimes you’ll only need to type the name of a command, and sometimes you’ll need to include arguments after the name. An argument is an additional piece of information that changes the command’s behavior.

Here are three examples using the ls command:

  • ls: lists contents of working directory; no arguments
  • ls Desktop: lists contents of the Desktop directory; Desktop is an argument
  • ls -a Desktop: lists contents of the Desktop directory, including hidden files; -a and Desktop are arguments

The third command includes a special type of argument called a flag. A flag normally starts with one or two dashes, and it enables a setting in the command. The -a flag tells ls to include hidden files in its output.

Spaces

Spaces separate each part of a command (the name and arguments). This can cause problems if you use spaces in your filenames. For example, if you have a folder named My Folder, then the command ls My Folder will not list the contents of My Folder. It will instead try to list the contents of two directories: My and Folder. We can fix this by quoting the path (ls 'My Folder') or escaping the spaces (ls My\ Folder), but this is easy to forget and takes longer to type. If you avoid using spaces in filenames, then you don’t have to worry about this.

Command History and Tab Completion

Your terminal saves a history of every command you enter. You can go back through this history with the up and down arrow keys, which lets you repeat a command without retyping it.

The tab key will attempt to complete the path or command name you are currently typing. This saves a lot of typing AND helps you avoid mistakes. If you haven’t typed enough to narrow it down to one possibility, then you can hit tab again to see a list of every remaining option.

Compact Source Files

This is for students with previous Java experience who want an explanation of how our programs differ from what they’re used to writing. It may also be useful if you want to convert examples you find elsewhere to the format we’re using.

In this class, most of our programs will be written as Compact Source Files. You can read this JDK Enhancement Proposal if you want the details. For now, I’ll summarize the main points:

  • Java normally requires a lot of boilerplate to write even the simplest program. That boilerplate code involves Java features that a new programmer won’t be equipped to understand.
  • Basic input and output (I/O) operations, which most of our programs will need, also require features that are difficult for new programmers to understand.
  • A compact source file is a more concise way of writing a Java program. It removes most of the boilerplate from small programs and provides a few simple I/O functions. This lets us focus on the actual program we’re trying to write and introduce new concepts when we’re ready for them.

The rest of this guide will cover how to convert standard Java programs into compact Java programs.

1. Remove the Class

Standard Java programs will have most of their code inside of a class:

public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello World");
    }
}

We don’t need the class:

public static void main(String[] args) {
    System.out.println("Hello World");
}

2. Simplify Main

The main function is more complex in a standard Java program:

public static void main(String[] args) {
    System.out.println("Hello World");
}

All we need is void main():

void main() {
    System.out.println("Hello World");
}

3. Output

Standard Java programs use System.out. or System.err. followed by println, print, or printf for output:

System.out.println("Regular output.");
System.err.println("Error message.");

We use IO.println and IO.print for output:

IO.println("Regular output.");
IO.println("Error message.");

To finish converting the earlier Hello World example:

void main() {
    IO.println("Hello World");
}

4. Input

Standard Java programs need to create a special object (such as a scanner or buffered reader) to read user input:

java.util.Scanner stdin = new java.util.Scanner(System.in);

System.out.print("Please enter some input: ");
var input = stdin.nextLine();

System.out.println("Please enter more input: ");
var input2 = stdin.nextLine();

We use IO.readln with an optional prompt message:

var input = IO.readln("Please enter some input: ");
var input2 = IO.readln("Please enter more input: ");

Types and Syntax

Terms

  • variable: name assigned to a value
  • declare: create a new variable and set its type
  • assign: store a value in a variable
  • initialize: assign the first value to a new variable; uninitialized variables cannot be used for anything
  • literal: a “hard-coded” value; the value is written directly in the program
  • type: the kind of information represented by an expression or value, such as a number, text, or a complex object
  • expression: something our program can compute/evaluate; has a resulting type and value
  • statement: an instruction our program can perform
  • operator: a symbol representing an operation that can be performed on one or more values, such as addition or subtraction

also note:

  • expressions can be as simple as a single variable name or literal value, or it can consist of many expressions joined by function calls and operators
  • something can be both a statement and an expression

  • String: a sequence of characters (text)

  • double: a real number
  • boolean: true/false

  • literals

  • terms:
    • function: named block of code that we can call (run the code in the block)

Useful Functions and Methods

  • i/o functions:
    • IO.print(x) converts x into a String and prints it to standard output
    • IO.println(x) as IO.print, plus it adds a line break after x
    • IO.println() with no argument, IO.println just prints a line break
    • IO.readln(x) prints x (as IO.print), blocks (pauses program) until user presses enter, and returns whatever the user typed (uses standard input) before pressing enter as a String
  • parsing functions:
    • Integer.parseInt(x) attempts to convert a String x to an int
    • Double.parseDouble(x) attempts to convert a String x to a double
    • Boolean.parseBoolean(x) attempts to convert a String x to a boolean
    • these functions will crash the program if given a String that cannot be parsed: for example, Integer.parseInt("abc")
  • math functions:
    • Math.pow(x, y) returns x raised to the power y
    • Math.abs(x) returns the absolute value of x
    • Math.max(x, y) returns the largest of its two arguments
    • Math.min(x, y) returns the smallest of its two arguments
  • String methods:
    • assume x in the examples below is a variable of type String, but it can be replaced with any String variable, literal, or expression
    • x.toUpperCase() returns a String identical to x, but with every letter capitalized
    • x.toLowerCase() I bet you can’t guess what this one does
    • x.length() returns the number of characters in x
    • x.substring(y) returns a substring of x starting at index y
    • x.substring(y, z) returns a substring of x starting at index y and ending on the character before index z
    • a substring is a portion of a string: “ello W” is a substring of “Hello World”
    • an index is how far a character is from the start of a string: the letter “H” has an index of 0 because it’s at the start, but “W” has an index of 6 because it’s six characters away from the start

Defining Functions

  • functions can return a value of any type:
    • void (used for main and some earlier examples) means the function returns nothing
    • another type, such as int or String, means we can store the result of calling that function in a variable of that type
  • functions can declare parameters to accept input from the code that calls them:
    • a parameter is declared in the parenthesis after the name
    • multiple parameters are separated by commas
    • a parameter is declared like a variable: a type followed by an identifier
    • a parameter’s value is set by the code that calls the function, not the code in the function’s body
    • a parameter can (and should) be used in the function body

Calling Functions

  • if a function returns something (not void), then we can use it as an expression or a part of an expression
  • if a function has one or more parameters, then we have to include those parameters in the parenthesis after the function’s name whenever we call it

  • information entering and leaving a function
    • we do this with parameters and return values

why to do I/O at the program level:

  • input
    • we need the user to supply information
    • we never need to accept this information from other code
  • output
    • we need the user to see information
    • we never need other code to have the information

why to do I/O at the function level (and NOT at the program level):

  • we want a function to be flexible and able to interact with other code
    • we may need to accept information from other parts of the program
    • we may need to use the information in another part of the program
  • this is usually preferable to doing I/O at the program level

common beginner mistakes:

  • assigning to parameters in function body
  • using IO.readln instead of a parameter
  • using IO.println or IO.print instead of a return statement

three forms:

    // if block

    // if block

    // else block
}

    // if block 1

    // if block 2

    // if block 3

    // else block

  • simple if
    • condition is a boolean expression
    • code block executed if condition evaluates to true
    • code block skipped if condition evaluates to false
  • if-else
    • first code block (if block) executed if condition evaluates to true
    • second code block (else block) executed if condition evaluates to false
    • no condition after else, only after if
  • else ifs
    • only executes code block after the first if whose condition evaluates to true
    • skips all other blocks
    • an else block at the end will always execute when all conditions evaluate to false
    • ending with an else if instead of an else means we will skip all code blocks when all conditions evaluate to false

While Loops

just one form, just like a simple if:

while (condition) {
   // loop body
}

a while loop will repeat a code block or a single statement until its condition is false, which carries several implications:

  • unlike an if statement, a loop’s code block can execute multiple times
  • if the condition is false when we reach the loop, then the body will execute zero times
  • if the body does not eventually change the condition from true to false, then the loop will repeat indefinitely (an infinite loop)
    • you can use control + c (still control + c on macOS, not command + c) on the terminal to cancel a running program if you get stuck in an infinite loop

one useful pattern is a “counting loop” (see Count.java example)

a good default template for a counting loop is:

// default to starting at 0 unless you have a good reason to start at another
// number
int i = 0;

// if the loop should repeat 25 times, then our condition would be `i < 25`
while (i < _TOTAL_NUMBER_OF_REPETITIONS_) {
    // whatever tasks the loop should repeat go here
    // these tasks can and often should make use of your counter variable i

    // increment your counter variable at the end of the loop body
    i = i + 1;
}

Pseudorandom Number Generators (PRNGs)

  • video on several PRNG algorithms: https://www.youtube.com/watch?v=XDsYPXRCXAs
  • pseudorandom: appearing random, but not actually random
    • start with a “seed” value (just the initial number)
    • use math and bitwise operations to get a new number
    • this number is both the random value and the new seed
  • simple algorithm for a random double (see video):
    • use the formula f(x) = (x * A + B) % C
    • A, B, and C are constants, must have certain properties (see video or https://en.wikipedia.org/wiki/Linear_congruential_generator)
      • video showed a formula with A = 9301, B = 49297, and C = 233280
    • first update the seed value (seed = f(seed))
    • then return seed / C to scale the random number to be from 0.0 to 1.0
      • must divide by C as a double, not an int
  • the seed value
    • if your seed is always the same, then you’ll get the same “random” sequence every time you run the program
    • you can initialize the seed value using a random value from another source or a nonrandom value that will change every time you run the program
    • simple solution: use System.currentTimeMillis() and convert the returned long to an int (this is easy for an attacker to exploit, but we’re not concerned about making our example algorithm crytographically secure)

example:

// declare seed outside function so it's not reset every time we call random()
// usually not good to make "global" variables like this, but it's necessary for
// our prng function
int seed = 1;
double random() {
    seed = (seed * 9301 + 47297) % 233280;
    return seed / 233280.0;
}

Scope and Global Variables

  • scope: where is an identifier valid, or where does the variable exist
  • local variable: variable scope limited to code block where it was declared
  • parameter: parameter scope limited to its function
  • “global” variable: declared outside functions, valid/exists everywhere
    • technically not a global variable, but in our programs it looks and acts like one
  • smaller scope is better
    • much easier to understand a variable if it’s limited to a small bit of code
    • bugs related to parameters and local variables are easier to track down
    • global variables more error-prone: code that interacts with a global variable can affect any other part of the program that also interacts with that variable
  • avoid using global variables, especially for convenience
    • there’s usually a better way to do whatever you’re trying to do with the global variable
    • using the global variable just avoids learning the better way to do it
    • the larger your programs, the more problems global variables will cause

Encapsulation

  • encapsulation = data hiding, controlling which code can interact with other code
  • several purposes:
    • able to make guarantees about data, keep objects in a valid state
    • prevents accidental changes that would cause bugs
    • makes code simpler to reason about
    • separate public API from private implementation details, making it easier for other programmers to use your code correctly
  • access modifiers: public, protected, private, and “package private” (no modifier given)
  • reason to mention it now is to explain the public keyword
  • more relevant in larger code bases
  • for us:
    • we’re not trying to learn OOP and won’t benefit much from the strict encapsulation you’d practice in an OOP course
    • we’ll avoid using access modifiers most of the time except where strictly needed
    • this isn’t necessarily considered good practice in Java, and if you take other Java courses you’ll do things differently

Scanner

  • the System class provides three objects for I/O:
    • System.out prints to standard out, just like IO.println and IO.print
    • System.err prints to standard err (intended for error messages and can be filtered out by terminal commands that process output)
    • System.in reads from standard in, just like IO.readln, but it produces raw bytes rather than strings and is harder to use
  • using a Scanner to process input:
    • Scanner stdin = new Scanner(System.in); creates a scanner to read from System.in
    • stdin.nextLine(); reads a string from standard in, just like IO.readln(), but it doesn’t accept a prompt message
    • you only need the .nextLine() method to read input, and you can convert to other types with the parse functions we’ve been using
    • the scanner has other methods (.nextInt(), .nextDouble(), etc.) to read and parse input into specific types, but these can cause surprising behavior if used before calling .nextLine()

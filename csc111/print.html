Description

From the course catalog:

This course is a continuation of CSC 110. Topics include advanced data structures (linked lists, stacks, queues, trees, hash tables), recursion, abstract data types, introductory algorithms, and intermediate object-oriented programming.

Objectives

CSC111 and CSC111L share objectives. Upon successful completion of both courses, students should be able to:

  1. Design, write, document, and test sophisticated programs using an object-oriented language.
  2. Identify situations where recursion is an appropriate problem solving strategy.
  3. Compare array and linked data structures and implement problem solutions using each.
  4. Describe the characteristics of stack and queue data structures and implement problem solutions using each.
  5. Describe characteristics of tree data structures and implement problem solutions using them.
  6. Explain search and sorting methods and implement programs using each.
  7. Describe factors that determine algorithm efficiency and measure the efficiency of searching and sorting algorithms.

ABET (CAC) - Student Outcomes Addressed:

  • C1) Analyze a complex computing problem and to apply principles of computing and other relevant disciplines to identify solutions. (Addressed by Objectives 1-7)
  • C2) Design, implement, and evaluate a computing-based solution to meet a given set of computing requirements in the context of the program’s discipline. (Addressed by Objectives 1,7)
  • C3) Communicate effectively in a variety of professional contexts. (Addressed by Objectives 1)
  • C6) Apply computer science theory and software development fundamentals to produce computing-based solutions. (Addressed by Objectives 1-7)

Grading

The grading system for my classes is loosely based on labor-based grading. It might be a bit different from what you’re used to, but it should be easy enough to understand:

  • Assignments are graded as follows:
    • An assignment is broken into one or more units of work, which I’ll refer to as tasks.
    • Each task has a point value and is marked either complete (you get 100% of the task’s points) or incomplete (you get 0 points for the task) when your work is graded. There is no partial credit for incomplete tasks.
    • Most assignments allow multiple submissions, and only your best submission counts.
  • Your final grade is the sum (not the average) of your assignment grades and uses the letter grade scale in QU’s grading system. This means five points on an assignment is equal to five points on your final grade.
  • There are no exams. Your final grade is only determined by your assignment grades.

Assignments

Our assignments fall into two categories:

  1. Autograded: these assignments are graded by a set of automated tests. A task must pass all of its tests to count as complete, and any requirements being tested are either stated in the assignment or revealed in the feedback after you pass certain tests. They cover most of the topics in our course objectives and make up the bulk of your grade.
  2. Other: there will be at least one larger open-ended assignment, and there may be other small assignments or optional tasks that will earn you a few extra points.

Feedback

You will receive a link to a shared OneDrive folder near the beginning of the semester, which you should star or bookmark. I will upload all assignment feedback to that folder. Expect feedback on autograded assignments by the end of the day every Monday, Wednesday, and Friday.

Most feedback I give will be automatically generated by my tests. You should review this feedback yourself, but feel free to ask me if you have questions about it. I automate parts of the grading process so that I can handle large numbers of submissions and revisions, not because I don’t want to discuss your code with you. The automation gives me more time to work with people who have questions or need extra help. It also allows for faster feedback and more opportunities for you to revise work.

Academic Integrity and Accountability

This policy covers is specific to our class and expands on the following QU policies:

  • Academic Integrity and Accountability (AIA): https://catalog.qu.edu/university-policies/academic-integrity-policy/
  • Artificial Intelligence (AI): https://provost.qu.edu/artificial-intelligence/

Everything in the above policies still applies to our class except as noted below.

Exceptions

An assignment may include rules not mentioned in this policy or allow exceptions to the rules in this policy. Rules specific to an assignment always take precedence over the more general rules in this policy. These will be noted in an AIA section near the beginning of the assignment.

You may also be able to do something normally banned by this policy if you ask for and receive my written permission first. For example, I may approve using AI-generated code on an assignment if it serves an educational purpose and doesn’t undermine the goals of the assignment.

Artificial Intelligence

The use of any “AI” tools (chatbots, plugins, assistants, etc.) on assignments is prohibited.

Unauthorized Possession of Materials

If you have not completed an assignment, then you must not at any point access or possess a full or partial solution to that assignment. This includes material shared by a classmate, posted online, or generated by an AI.

AI Use and Plagiarism

The following are not allowed and will be reported as a violation of the AIA policy:

  • Copying, paraphrasing, or otherwise basing part of your work on code you did not write. This includes AI-generated code, code found online, and code written by a classmate.
  • Asking or allowing someone else to write code for you, dictate code to you, or in some other way do part of your work for you.
  • Splitting work between multiple people or otherwise working together on an assignment that doesn’t explicitly allow group work.
  • Using AI plugins in your code editor or using a code editor with AI features. If you choose not to use the recommended software, then you are responsible for evaluating your editor and ensuring it doesn’t violate this policy. Do not ask me to evaluate an editor for you.

The following are allowed:

  • Using material I provide you. If I include code in an assignment, show you code during class/office hours, or share code in my notes or other material, then you’re free to use it in any assignments for this class.
  • Asking other people for help or looking up online sources for help. However, your use of other sources and the help you receive from other people must not violate any other rules in this policy. Any assistance should focus on helping you understand the problem and the relevant programming concepts, not on solving the problem or writing the code.
  • Asking someone for assistance with debugging (fixing an error or determining why your program didn’t work correctly). This assistance should be limited to helping you identify and explain the problem (for example: what does a particular error mean, how can you further test your code to locate a problem, why isn’t the program producing the correct result). Determining how to fix your code and making those changes is your job, and nobody else should be doing this for you.

Sharing Work

Sharing work you did for this class is allowed in most situations. For example, you’re welcome to include your completed assignments in public portfolios, share them with people outside the class for feedback, or use them as the basis for other projects.

Sharing work is unacceptable in the following situations:

  • You may not post work online in a way that explicitly encourages AIA violations or implies it is meant for students to copy or otherwise use to violate the AIA policy.
  • You may not share work with another person for the purpose of facilitating an AIA violation. This includes sharing work with a classmate you know has not completed the assignment.

Terminal

A terminal provides a text-based interface for running commands on your computer. These commands are executed by another program called a shell, such as bash or zsh.

We will be using the terminal almost every class to interact with files and run the programs we write. You should have a terminal and shell that can run common bash commands so that you can follow along with my examples. Linux and macOS include such programs by default, but Windows users will need to install one.

Recommended Software

  • Windows: use the Git Bash terminal, which will be included when you install git.
  • macOS and Linux: use the default terminal.

Using a Terminal

We’ll cover this in class, but here’s a brief introduction so that you can use the terminal to check your Java installation:

  1. Find and run your terminal program by typing its name in your computer’s search bar. On Windows you should be using “Git Bash” (make sure you install git first) and on macOS or Linux the program will just be called “Terminal”.

Java

This course relies on new features that were previewed starting in Java 21 and finalized in Java 25. You need to install the JDK (Java Development Kit) for Java 25 or later to access the correct version of these features.

Installation

The Eclipse Foundation provides free builds of the JDK, which you can download from their releases page. That page should default you to the tab for “JDK 25 - LTS”. It should also have download buttons for macOS (a .PKG file for arm64 Macs) and Windows (a .MSI file for x86_64 Windows computers) right below those tabs. If you’re using either of those operating systems, then download and run the appropriate installer. If you need help with the installers: Windows guide, Mac guide.

On Linux, you should check your package manager to see whether it has version 25 of Java. If not, then you can manually install the .TAR.GZ download. Please see me during office hours if you need help with this.

Check Version

After you install java, make sure you’ve installed the correct version by running both of the following commands on your terminal:

java -version
javac -version

Both commands should show a version number starting with 25. If either command shows an error or a different version number, then there is a problem with your installation. If your terminal was open during the installation, then restart your terminal and try the commands again.

Troubleshooting

Any of the following can cause problems. Before you ask for help, make sure you…

  • downloaded the correct jdk (not jre) installer for your operating system
  • did not download or run any other installers
  • restarted your terminal before checking the version

If you downloaded or ran multiple installers, then uninstall (see below) all incorrect versions and delete the incorrect installers. Then download and run the correct installer, restart your terminal, and check the version again.

Uninstall

If you install the wrong version or want to upgrade to a newer version, then you should first uninstall the old version.

On Windows, you can follow these directions from Microsoft. On Mac, you will need to delete the installation from your /Library/Java/JavaVirtualMachines folder using the directions below. Be careful when accessing the /Library folder! Your OS normally hides this folder from you because deleting or modifying the wrong files can cause problems with your computer. Only delete the folder for the Temurin installation you want to get rid of.

  1. Open the Go to Folder dialog in Finder.
  2. Copy /Library/Java/JavaVirtualMachines into the search bar and open that folder. Make sure you copy this path exactly as shown, including the leading /. There are other folders named Library on a Mac, so taking a different shortcut may lead you to one that doesn’t contain the Java installation.
  3. You should see a folder for each version of the Temurin JDK you’ve installed. Each folder’s name should be in the format temurin-<version>.<jdk or jre>. For example, the Java 25 JDK might be named temurin-25.0.1+8.jdk. Delete the unwanted installations. You may need to enter your password to delete the folder.

Editor

A text editor is a program that edits plain text files. Most code is written as plain text with a file extension indicating which programming language it represents. Text editors are different from programs like Microsoft Word or Google Docs, which edit formatted text and should not be used to write or edit code.

Writing code is obviously an important part of this class, so you’ll need a text editor. I recommend using Geany, but if you’re feeling adventurous then you might want to try vim.

You may also prefer an Integrated Development Environment (IDE), which is essentially an editor with additional features like error checking and code completion. Most editors have some IDE features, but an IDE like Eclipse has far more features than Geany or an unconfigured vim. I don’t plan to give instructions for Eclipse, but it’s worth checking out if you want something more powerful than Geany.

I also want to recommend not using VS Code as your editor. It has some AI code generation features that may be enabled by default, and using them violates this class’s AIA policy.

Vim and Neovim

Vi, Vim, and Neovim are a group of related editors that can run directly on your terminal without a graphical interface. These editors are designed to be fast and efficient, but you need to learn and use their keyboard shortcuts instead of the mouse to get the most out of them.

I personally use neovim and prefer this style of editor over graphical editors like Geany. However, I don’t recommend vim/neovim to most people because they can be difficult and frustrating to learn, especially for beginners who are used to relying on a mouse. I think the power and customizability of vim and similar editors is well worth the effort required to learn them, but most people prefer graphical editors.

If you’re curious, then give vim or neovim a try. You can also look into emacs, which is not related to vim but has a similar focus on keyboard shortcuts.

You can consider these “semi-recommended” programs. I can provide some assistance with vim or neovim during office hours, but you should be ready to learn them on your own. There are plenty of resources available for teaching yourself to use these programs, even games!

Git

Terminal Guide

We’ll talk about most of this in class, but I wrote this guide a while ago and have included it as a reference in case you want to review this material. We use the terminal constantly, so it helps to know this stuff.

Commands

You interact with a terminal by typing a command and reading the command’s output. This section will cover the structure of a command and some terms you’ll need to know to understand these commands.

There are differences between Windows and Unix-like systems, but most of these differences won’t matter if you are using a bash emulator on Windows. For example, Windows uses a backslash to separate directory names in paths, but your bash emulator should allow you to use a forward slash as if you were on a Mac or Linux computer. My notes focus on Unix-like systems, but I will try to note differences between Git Bash and bash/zsh when they come up.

Terms

Some terms that will be used to explain terminal commands. Refer back to this list if you come across something you don’t understand.

  • Directory: another word for a folder on your hard drive. A directory can contain files and other directories.
    • Your working directory is the directory in which your terminal (or another program) is currently running. Your terminal has direct access to files in its working directory.
    • Your home directory is the where the terminal starts. This should be your user folder: /home/username on Linux, /Users/username on Mac, and C:\Users\username (displayed as /c/Users/username in a bash emulator) on Windows.
  • Path: a path is the location of a file or directory. It consists of directory names separated by forward slashes (/). The path ends with the name of the file or directory it leads to.
    • An absolute path tells you how to reach a file or directory from the root of a file system. An absolute path starts with a forward slash on Unix-like systems. For example, the absolute path to my Desktop can be written as /home/zach/Desktop or /home/zach/Desktop/. The forward slash at the end of the path is optional, and it can only be included when the path points to a directory (such as the Desktop folder).
    • A relative path tells you how to get from one location (usually your working directory) to a specific file or directory. For example, if I were in the directory /home/zach and wanted to reach /home/zach/Desktop/notes.txt, then the relative path Desktop/notes.txt tells me how to reach that file from my current location.
    • Your computer also has a list of paths that it checks whenever you ask it to run a program. This list is stored under the label “path” or “PATH”, and we can just refer to it as “the path”. When you install a new program, the installer will usually add its location to the path or create a shortcut in a directory that was already on the path. If a program is on the path, then you only have to type its name to run it. If a program is not on the path, then you have to type a path to that program to run it.

Command Structure

Commands on the terminal can refer to programs or built-in functions. Sometimes you’ll only need to type the name of a command, and sometimes you’ll need to include arguments after the name. An argument is an additional piece of information that changes the command’s behavior.

Here are three examples using the ls command:

  • ls: lists contents of working directory; no arguments
  • ls Desktop: lists contents of the Desktop directory; Desktop is an argument
  • ls -a Desktop: lists contents of the Desktop directory, including hidden files; -a and Desktop are arguments

The third command includes a special type of argument called a flag. A flag normally starts with one or two dashes, and it enables a setting in the command. The -a flag tells ls to include hidden files in its output.

Spaces

Spaces separate each part of a command (the name and arguments). This can cause problems if you use spaces in your filenames. For example, if you have a folder named My Folder, then the command ls My Folder will not list the contents of My Folder. It will instead try to list the contents of two directories: My and Folder. We can fix this by quoting the path (ls 'My Folder') or escaping the spaces (ls My\ Folder), but this is easy to forget and takes longer to type. If you avoid using spaces in filenames, then you don’t have to worry about this.

Command History and Tab Completion

Your terminal saves a history of every command you enter. You can go back through this history with the up and down arrow keys, which lets you repeat a command without retyping it.

The tab key will attempt to complete the path or command name you are currently typing. This saves a lot of typing AND helps you avoid mistakes. If you haven’t typed enough to narrow it down to one possibility, then you can hit tab again to see a list of every remaining option.

Examples

If I want to show you what it looks like when you run a terminal command, then I will include both the command and its output. It might look something like this:

$ ls -a ~/Desktop
. .. todo.txt lesson_plan.md

Terminals display a prompt when you are allowed to type a command. This prompt can include your username and working directory, and it usually ends in a dollar sign, percent sign, or other special character. When you type a command, it will appear after that special character. For example, the prompt on my work desktop terminal looks like this before I type a command:

zach@zach-work-desktop:~$

If I type the command ls -a ~/Desktop, then it will appear after the prompt:

zach@zach-work-desktop:~$ ls -a ~/Desktop

We usually use a lone dollar sign at the start of a line to abbreviate this prompt, but it is not part of the command. For example, if you wanted to imitate the example below, then you would only type ls -a ~/Desktop on your terminal and press the enter key.

$ ls -a ~/Desktop
. .. todo.txt lesson_plan.md

Here’s a longer example involving five terminal commands, two of which produce output:

$ cd Desktop/csc110
bash: cd: csc110: No such file or directory
$ cd Desktop
$ mkdir csc110
$ ls
csc110
$ cd csc110

In the above example, I ran the commands cd Desktop/csc110, cd Desktop, mkdir csc110, ls, and cd csc110. The messages bash: cd: csc110: No such file or directory and csc110 were the output of two of those commands.

Navigation

Before you can use the terminal to run your programs, you need to know how to navigate your filesystem and find those programs. This is essential for doing anything in the terminal. The following three commands will help:

  • pwd (print working directory): tells you where you are
  • cd (change directory): moves the terminal to a new folder
  • ls (list): lists files and folders in your current location

pwd

Run pwd to print the absolute path of your working directory. You won’t need to use this command often, but it’s useful if you aren’t sure exactly where you are.

cd

Running cd with no arguments will take you to your home directory. Running cd followed by a directory’s path will take you to that directory. You can use a relative or absolute path.

For example:

  • If I’m in the directory /home/zach/, which contains the directory Documents, then I can move into Documents with cd Documents or cd /home/zach/Documents/.
  • If I’m in /home/, then I can move into Documents with cd zach/Documents or cd /home/zach/Documents/, but not cd Documents.

Finally, there are a few special names that act as shortcuts for commonly-used directories:

  • ~ Refers to your home directory. In the above example, I could reach my Documents folder from any location with cd ~/Documents.
  • . Refers to the working directory. You won’t need to use this one right now, but you may see it in some paths or after running the ls -a command.
  • .. Refers to the working directory’s parent directory. This is useful if you want to go back to a directory that appears earlier in your path. For example, if your working directory is /home/zach/Documents/csc110/homework/, then .. refers to /home/zach/Documents/csc110/. You can use .. multiple times to go back further. In the previous example, ../.. would refer to /home/zach/Documents/, and cd ../.. would move me to that directory.

ls

Running ls with no arguments will list the contents of your working directory. Running ls followed by a path will list the contents of that directory (if the path points to a directory) or the name of that file (if the path points to a file).

This command is helpful if you’re looking for a file or want to see what’s in your current location.

The -a flag mentioned earlier will show hidden files, and the -l flag will show more details about the files.

public class Example3 {
    public static void main(String[] args) {
        // if there are no program args, then the program ends!
        if (args.length == 0) {
            System.out.println("Sorry, I need program args.");
            // ends the program early
            // exiting with a nonzero value indicates to other programs that
            // something went wrong
            System.exit(1);
        }

        // check whether there are an even or odd number of args
        if (args.length % 2 == 0) {
            System.out.println("Even arg count detected!");
            // call another static method
            even(args);
        } else {
            System.out.println("Odd arg count detected!");
            // call another static method
            odd(args);
        }
    }

    public static void even(String[] args) {
        // print only the even-numbered args
        for (int i = 1; i < args.length; i += 2) {
            // our target audience is non-programmers, and their brains aren't
            // zero-indexed
            System.out.printf("%d. %s\n", i + 1, args[i]);
        }
    }

    public static void odd(String[] args) {
        // print only the characters encoded as odd numbers
        // why are we doing that? hey, look! an enhanced for loop!
        for (String arg : args) {
            // wow, a nested loop!
            for (int i = 0; i < arg.length(); i++) {
                // get character
                char c = arg.charAt(i);
                // checking if odd
                if (c % 2 != 0) {
                    System.out.print(c);
                }
            }
            // line break after each arg
            System.out.println();
        }
    }
}

Encapsulation

  • encapsulation
    • hiding data from parts of the program that should not access it
    • allows classes to guarantee a valid internal state, maintain invariants
    • only expose the public API necessary to use the class
  • access modifiers
    • public
    • private
    • protected
    • none
  • invariant: something that must always be true
  • example: immutable Strings
    • internal char array is private
    • strings cannot change after creation
  • by default, instance variables are private

Packages

  • namespacing: separating code into “namespaces” allows name reuse without conflict
    • C# has a namespace keyword
    • Java has packages (declared with package keyword)
    • classes in both languages can also act as namespaces
  • packages must match directory structure
    • a project has a root directory (typically wherever we are when running the java command)
    • files in root declare no package and are in the “default package”
    • a file abc/def/Example.java would declare the package abc.def
  • when you import from standard library or another library, your import includes the package
    • for example: Scanner is in the package java.util
  • won’t need to deal with packages on your end too much this semester
  • unless told otherwise, your code should just use the default package (i.e. don’t declare a package at all)

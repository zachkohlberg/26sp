Grading

The grading system for my classes is loosely based on labor-based grading. It might be a bit different from what you’re used to, but it should be easy enough to understand:

  • Assignments are graded as follows:
    • An assignment is broken into one or more units of work, which I’ll refer to as tasks.
    • Each task has a point value and is marked either complete (you get 100% of the task’s points) or incomplete (you get 0 points for the task) when your work is graded. There is no partial credit for incomplete tasks.
    • Most assignments allow multiple submissions, and only your best submission counts.
  • Your final grade is the sum (not the average) of your assignment grades and uses the letter grade scale in QU’s grading system. This means five points on an assignment is equal to five points on your final grade.
  • There are no exams. Your final grade is only determined by your assignment grades.

Commands

You interact with a terminal by typing a command and reading the command’s output. This section will cover the structure of a command and some terms you’ll need to know to understand these commands.

There are differences between Windows and Unix-like systems, but most of these differences won’t matter if you are using a bash emulator on Windows. For example, Windows uses a backslash to separate directory names in paths, but your bash emulator should allow you to use a forward slash as if you were on a Mac or Linux computer. My notes focus on Unix-like systems, but I will try to note differences between Git Bash and bash/zsh when they come up.

Terms

Some terms that will be used to explain terminal commands. Refer back to this list if you come across something you don’t understand.

  • Directory: another word for a folder on your hard drive. A directory can contain files and other directories.
    • Your working directory is the directory in which your terminal (or another program) is currently running. Your terminal has direct access to files in its working directory.
    • Your home directory is the where the terminal starts. This should be your user folder: /home/username on Linux, /Users/username on Mac, and C:\Users\username (displayed as /c/Users/username in a bash emulator) on Windows.
  • Path: a path is the location of a file or directory. It consists of directory names separated by forward slashes (/). The path ends with the name of the file or directory it leads to.
    • An absolute path tells you how to reach a file or directory from the root of a file system. An absolute path starts with a forward slash on Unix-like systems. For example, the absolute path to my Desktop can be written as /home/zach/Desktop or /home/zach/Desktop/.     • A relative path tells you how to get from one location (usually your working directory) to a specific file or directory.     • Your computer also has a list of paths that it checks whenever you ask it to run a program. This list is stored under the label “path” or “PATH”, and we can just refer to it as “the path”. When you install a new program, the installer will usually add its location to the path or create a shortcut in a directory that was already on the path. If a program is on the path, then you only have to type its name to run it. If a program is not on the path, then you have to type a path to that program to run it.

Command Structure

Commands on the terminal can refer to programs or built-in functions. Sometimes you’ll only need to type the name of a command, and sometimes you’ll need to include arguments after the name. An argument is an additional piece of information that changes the command’s behavior.

Here are three examples using the ls command:

  • ls: lists contents of working directory; no arguments
  • ls Desktop: lists contents of the Desktop directory; Desktop is an argument
  • ls -a Desktop: lists contents of the Desktop directory, including hidden files; -a and Desktop are arguments

The third command includes a special type of argument called a flag. A flag normally starts with one or two dashes, and it enables a setting in the command. The -a flag tells ls to include hidden files in its output.

Spaces

Spaces separate each part of a command (the name and arguments). This can cause problems if you use spaces in your filenames. For example, if you have a folder named My Folder, then the command ls My Folder will not list the contents of My Folder. It will instead try to list the contents of two directories: My and Folder. We can fix this by quoting the path (ls 'My Folder') or escaping the spaces (ls My\ Folder), but this is easy to forget and takes longer to type. If you avoid using spaces in filenames, then you don’t have to worry about this.

Command History and Tab Completion

Your terminal saves a history of every command you enter. You can go back through this history with the up and down arrow keys, which lets you repeat a command without retyping it.

The tab key will attempt to complete the path or command name you are currently typing. This saves a lot of typing AND helps you avoid mistakes. If you haven’t typed enough to narrow it down to one possibility, then you can hit tab again to see a list of every remaining option.

  • positions containing the space character ' ' are considered empty
  • positions containing any other character are considered occupied, and we cannot enter them

  • from our current position, we are allowed to move to the adjacent north, south, east, or west position as long as it is empty
    • for example, we could move from (7, 3) to (7, 4), which is one cell south, but not to (7, 5) (two cells south) or (8, 4) (one cell diagonally southeast)

                throw new RuntimeException(
                        String.format(
                                "array[%d] (%d) is greater than array[%d] (%d)",
                                i - 1, array[i - 1], i, array[i]));
            }
        }

        for (int i = start; i < end; i++) {

                throw new RuntimeException(

                                i, array[i], i, original[i]));
            }

    // same elements as original. The order of the elements does not matter.
    public static void assertSameContents(int[] array, int[] original, int start, int end) {
        // The hash maps are used to track how many times each integer appears
        // in this section of array and original. We can't assume anything about
        // the order of these elements, so we count everything first and then
        // compare the counts.
        HashMap<Integer, Integer> arrayCounts = new HashMap<>(), originalCounts = new HashMap<>();

        for (int i = start; i < end; i++) {
            arrayCounts.put(array[i], arrayCounts.getOrDefault(array[i], 0) + 1);
            originalCounts.put(original[i], originalCounts.getOrDefault(original[i], 0) + 1);
        }

        if (!arrayCounts.equals(originalCounts)) {
            throw new RuntimeException("array and original don't contain the same set of elements");
        }

Note

In my pseudocode’s for loops, I use “to” to imply a value is included in a range and “until” to imply a value is excluded from the range. Functions that take a start and end parameter to specify a range will consider the start to be inclusive and the end to be exclusive.

bubbleSort(array, start, end):
    for each lastSwapIndex from end - 2 until start
        for each i from start to lastSwapIndex
            if array[i] > array[i + 1]
                swap array, i, i+1

selectionSort(array, start, end):
    for each swapIndex from end - 1 to start
        greatestElementIndex = start
        for each i from start + 1 to swapIndex
            if array[i] > array[greatestElementIndex]
                greatestElementIndex = i
        swap array, greatestElementIndex, swapIndex

insertionSort(array, start, end):
    for each sortedEnd from start + 1 until end
        for each i from sortedEnd until start
            if array[i] < array[i - 1]
                swap array, i, i-1
            else
                break

        return
    sort(array, start, (end + start) / 2)
    sort(array, (end + start) / 2, end)
    merge(array, start, end)

merge(array, start, end):
    mergeArray = new array with length of (end - start)
    mid = (start + end) / 2
    index1 = start
    end1 = mid
    index2 = mid
    end2 = end

    for each i from 0 until mergeArray.length
        if index1 < end1 and index2 < end2
            if array[index2] < array[index1]
                copy array[index2] to mergeArray[i] (counts as a swap)
                increment index2
            else
                copy array[index1] to mergeArray[i] (counts as a swap)
                increment index1
        else if index1 < end1
            copy array[index1] to mergeArray[i] (counts as a swap)
            increment index1
        else
            copy array[index2] to mergeArray[i] (counts as a swap)

Implement a quicksort algorithm (Quicksort.java). Same requirements as Task 1.

Pseudocode

quicksort(array, start, end):
    if end - start < 2
        return
    pivot = partition(array, start, end)
    quicksort(array, start, pivot)
    quicksort(array, pivot + 1, end)

partition(array, start, end):
    pivot = random number from start (inclusive) to end (exclusive)
    swap array, start, pivot

    secondPartitionStart = start + 1
    for each i from start + 1 until end
        if array[i] < array[start]
            swap array, i, secondPartitionStart
            increment secondPartitionStart

    pivot = secondPartitionStart - 1
    swap array, start, pivot
    return pivot

    }

        if (newRadius < 0) {
            this.radius = 0;
        } else {
            this.radius = newRadius;
        }
    }

        return this.x;
    }

        return this.y;
    }

Recursion Intro

what is recursion?

  • a recursive function is a function that calls itself
  • a recursive algorithm appears in its own steps

why learn/use it?

  • to repeat something, like a loop
  • compared to iterative (loop) version, recursive algorithms can be simple and concise
  • more powerful than plain loops and nested loops
  • some problems are easier to understand through recursive algorithms, even if you end up converting it to an iterative algorithm
  • important in functional programming, math (have you seen proof by induction yet?)

why not use it?

  • expensive: function call overhead compared to loop overhead
    • some compilers can optimize recursive functions into loops
  • loops are equally powerful when combined with a stack

  1. use recursive functions to perform iteration, imitate for loops (today)
  2. write more complex recursive functions, aware of branching/backtracking and the call stack (next class, maybe also next week)

The Stack and Infinite Recursion

the stack

  • region of memory in a running program
  • local variables and function calls use the stack
  • objects live in another region called the heap
  • calling a function creates a new “stack frame” where its parameters and local variables live
  • returning from a function frees that stack frame and passes return value to the stack frame below it

why this matters for recursion

  • recursion involves repeated function calls, which use the stack
  • recursion is powerful because it uses the stack to remember all the function calls it hasn’t finished resolving
  • we can (and will) make our own stack data structure, and a loop can use it to remember unfinished tasks like a recursive function
  • stack memory is limited, which leads to…

stack overflow

  • why can infinite loops run indefinitely?
    • each iteration finishes before the next starts
    • memory used in previous iteration is normally free for the next to use
    • the amount of memory needed does not change as the loop runs
  • why can’t infinite recursion run indefinitely?
    • wind-up phase consumes more stack memory with each function call
    • memory isn’t freed until function returns (unwind phase)
    • infinite wind-up will run out of memory before the unwind phase frees it

Stack Visualization

what is the stack?

  • program’s memory divided into:
    • read-only (code, constants)
    • stack (local variables, function calls)
    • heap (objects)
  • calling a function creates a new “stack frame”:
    • stores local variables, the state of this function call
    • if we call another function, then we return to this frame when the other call returns
    • this stack frame is freed when this function call returns

recursion’s power comes from the stack

  • implicitly uses stack memory to store an arbitrary number of unfinished function calls
  • this allows branching to an arbitrary depth (each function call may lead to another series of function calls, like a loop that can dynamically create a new nested loop)
  • loops have limited depth, tied to the number of nested loops
  • when we make our own stack, we can use it and a loop to implement recursive algorithms without recursive functions

winding and unwinding phases

  • room counting activity
  • factorial

box trace: https://homepages.uc.edu/~jonesbr/boxtrace.shtml

  • winding phase builds up a graph of function calls

Infinite Recursion

stack memory is limited, which limits recursion

  • when stack memory runs out, you get a StackOverflowError
  • normal recursive functions can do this if their depth is too great
  • infinite recursion will cause this if the compiler doesn’t optimize it into a loop

what causes infinite recursion?

  • no base case
  • recursively solving the same or larger problem instead of smaller problem (never reach base case)
  • shrinking the problem in a way that does not lead to a base case (may need an additional base case)

just like loops, you need to design recursive functions so that they always terminate

Search Algorithms

  • linear search
    • check each element starting at beginning
    • stop when target found or end of array
  • binary search
    • requirement: array is sorted
    • check middle element
      • if target, stop
      • if greater than target, binary search first half of array
      • if less than target, binary search second half of array
    • stop if the array (or section we’re searching) is empty

Practice: Guess a Number

binary search is the optimal strategy (wins in the fewest queries) for the following game:

  1. I choose a number from 1 to 1 billion.
  2. You pick a number, and I’ll tell you whether it’s less than my number.
  3. You can either guess my number or return to step 2. You only get one guess!

The GuessBot program below plays this game using a linear search strategy, which is much worse than binary search. Rewrite GuessBot to use a binary search!

/**
 * Implements an algorithm to guess a secret number. This starter version performs a linear search,
 * but you can improve it by switching to a binary search.
 *
 * @author Zach Kohlberg
 * @author YOU, after you rewrite this!
 */
public class GuessBot {
    public static void main(String[] args) {
        // choose a secret number from 1 to 1 billion
        SecretNumber secret = new SecretNumber();

        // shows what secret number will tell us before guessing
        System.out.println("Secret number:");
        System.out.println(secret);

        // use some algorithm to figure out the number
        int guess = findSecret(secret);

        // guess the number once we've figured it out
        secret.guessSecret(guess);

        // shows whether we guessed correctly
        // we will ALWAYS guess correctly if we correctly implemented our algorithm
        // we want to guess correctly with the fewest possible queries
        System.out.println("Secret number:");
        System.out.println(secret);
    }

    static int findSecret(SecretNumber secret) {
        // TODO: replace this inefficient linear search strategy (which takes about 500 million
        // queries on average) with a much faster binary search

        // try every possible value until we find the number
        for (int i = SecretNumber.MIN_VALUE; i <= SecretNumber.MAX_VALUE; ++i) {
            // we started with the lowest possible value, so we've reached the secret number as soon
            // as the secret isn't greater than our value
            if (!secret.isGreaterThan(i)) {
                return i;
            }
        }

        // the secret number always exists, and we should never fail to find it if we implement our
        // algorithm correctly
        throw new RuntimeException(
                "Our algorithm didn't find the secret. This means our code is wrong!");
    }
}

 * <p>I felt like using javadoc comments in this code, which is why you see some html tags. You can
 * use the javadoc program (included with your jdk installation) to generate a documentation website
 * for this class.
 *
 * @author Zach Kohlberg

public class SecretNumber {
    /** Lowest number allowed. */
    public static final int MIN_VALUE = 1;

    /** Highest number allowed. */
    public static final int MAX_VALUE = 1_000_000_000;

    private static final int NULL_GUESS = MIN_VALUE - 1;

    private int number;
    private int guess;
    private int queryCount;

    public SecretNumber() {
        number = (int) (Math.random() * MAX_VALUE) + MIN_VALUE;
        guess = NULL_GUESS;
        queryCount = 0;
    }

    /**
     * Used to check whether the secret value is greater than another number. Only accepts numbers
     * between the min and max values, and only allowed before guessing.
     *
     * @param n the number to compare to the secret number
     * @return true if the secret number is greater than n and false if it is not
     */
    public boolean isGreaterThan(int n) {
        if (guess != NULL_GUESS) {
            throw new SecretNumberException("You must not query after guessing.");
        }

        if (n < MIN_VALUE || n > MAX_VALUE) {
            throw new SecretNumberException(
                    String.format(
                            "You must only use numbers from %d to %d.", MIN_VALUE, MAX_VALUE));
        }

        ++queryCount;

        return number > n;
    }

    /**
     * Used to guess the secret number. Can only be called once, and only accepts a number between
     * the min and max value. This method should not be called until your code has narrowed the
     * secret down to only one possibility.
     *
     * @param n the guess
     * @return true if the guess is correct and false if it is not
     */
    public boolean guessSecret(int n) {
        if (guess != NULL_GUESS) {
            throw new SecretNumberException("You must not guess more than once.");
        }

        if (n < MIN_VALUE || n > MAX_VALUE) {
            throw new SecretNumberException(
                    String.format(
                            "You must only use numbers from %d to %d.", MIN_VALUE, MAX_VALUE));
        }

        guess = n;

        return number == guess;
    }

    @Override
    public String toString() {
        if (guess == 0) {
            return String.format("Guess: NONE, Queries: %d, Number: SECRET", queryCount);
        } else {
            return String.format(
                    "YOU %s! Guess: %d, Queries: %d, Number: %d",
                    number == guess ? "WIN" : "LOSE", guess, queryCount, number);
        }
    }

    public static class SecretNumberException extends RuntimeException {
        public SecretNumberException(String msg) {
            super(msg);
        }
    }
}

Sorting Game

Choose one person at your table to be the computer. Everyone else is a player.

The computer’s job is to run the game and manage the cards. The computer is neutral and should not try to help or interfere with the players.

The players have one goal: sort the cards without looking at them. The cards should be sorted from least to greatest when read from left to right. All players should sit on the same side of the table so there’s no confusion over the correct order.

  1. The computer shuffles six cards and deals them face down in a row.
  2. Choose a first player. All players are on the same team, but they will take turns going clockwise from the first player.
  3. On a player’s turn, they will choose two cards (don’t touch or look at them, just point to them so the computer knows which cards you chose).
    1. The computer looks at the chosen cards without showing them to any players and says which of the two cards is larger. All cards are different, so there won’t be any ties.
    2. The player chooses to swap the cards or leave them where they are.
    3. The player chooses to end the game (proceed to step 4) or continue on to the next player (repeat step 3 with that player). You should not end the game unless you believe the cards are sorted.
  4. Flip over each card. If the cards are sorted from least to greatest, then the players win! If not, then the players lose!

Sorting Algorithms Overview

basic operations:

  • comparison
  • swap

less common:

  • shift multiple elements instead of swapping (optimization for insertion sort)
  • create additional arrays and copy from one array to another (mergesort)
  • create more sophisticated data structures (heapsort, radix sort)

many algorithms, but we’ll look at five:

  • slower algorithms (n squared)
    • bubble sort
    • selection sort
    • insertion sort
  • faster algorithms (n log n)
    • mergesort
    • quicksort

other algorithms worth noting

  • robsort (http://www.robsort.org/) designed to be as inefficient as possible, O(n!)
  • radix sort (https://en.wikipedia.org/wiki/Radix_sort) sorts without comparing elements and can be very efficient for certain kinds of input

in practice

  • fast algorithms work well for large input sizes, but there’s overhead that makes them worse for small input sizes
  • the “slower” algorithms can outperform the fast algorithms for small input sizes
  • general-purpose sorting algorithms, like those in the Java standard library, will use a combination of sorting algorithms (for example, quicksort on large input sizes and insertion sort on small input sizes)

Comparison and Swap

Swapping elements in an array involves three steps, with a temporary variable to hold one of the elements:

swap(arr, i, j):
    temp = arr[i]
    arr[i] = arr[j]
    arr[j] = temp

You can write a swap method like the above pseudocode, or you can write those three steps whenever you need to perform a swap.

Comparing numbers is easy (just use the usual inequality operators). For more complex types, there are a few options. The simplest for strings is to use the compareTo method (https://docs.oracle.com/en/java/javase/25/docs/api/java.base/java/lang/String.html#compareTo(java.lang.String)). Calling a.compareTo(b) returns an int indicating one of three possible results:

  • a negative number means a is “less than” b
  • a positive number means a is “greater than” b
  • a zero means a and b are equal

Bubble Sort

sort(arr):
    for each lastSwap from arr.length - 2 to 0
        for each i from 0 to lastSwap
            if arr[i] > arr[i + 1]
                swap arr, i, i+1

Selection Sort

sort(arr):
    for each swapIndex from arr.length - 1 to 0
        greatest = 0
        for each i from 1 to swapIndex
            if arr[i] > arr[greatest]
                greatest = i
        swap arr, greatest, swapIndex

Insertion Sort

sort(arr):
    for each int index from 1 until arr.length
        for each i from index to 1
            if arr[i] < arr[i - 1]
                swap arr, i, i-1
            else
                break

Mergesort

sort(arr):
    mergeSort(arr, 0, arr.length)

mergeSort(arr, start, end):
    if end - start < 2
        return
    mergeSort(arr, start, (end + start) / 2)
    mergeSort(arr, (end + start) / 2, end)
    merge(arr, start, (end + start) / 2, end)

merge(arr, start, mid, end):
    mergeArr = new array with length of (end - start)
    index1 = start
    end1 = mid
    index2 = mid
    end2 = end

    for each i from 0 until mergeArr.length
        if index1 and index2 have not reached end of their section
            if arr[index2] < arr[index1]
                copy arr[index2] to mergeArr[i]
                increment index2
            else
                copy arr[index1] to mergeArr[i]
                increment index1
        else if index1 has not reached end of its section
            copy arr[index1] to mergeArr[i]
            increment index1
        else
            copy arr[index2] to mergeArr[i]
            increment index2

    copy mergeArr's contents (0 to length - 1) back to arr (start to end - 1)

Quicksort

sort(arr):
    quicksort(arr, 0, arr.length)

quicksort(arr, start, end):
    if end - start < 2
        return
    pivot = partition(arr, start, end)
    quicksort(arr, start, pivot)
    quicksort(arr, pivot + 1, end)

partition(arr, start, end):
    pivot = random number from start (inclusive) to end (exclusive)
    swap arr, start, pivot
    secondPartitionStart = start + 1
    for each i from start + 1 until end
        if arr[i] < arr[start]
            swap arr[i] and arr[secondPartitionStart]
            increment secondPartitionStart
    pivot = secondPartitionStart - 1
    swap arr, start, pivot
    return pivot

  • we’ll create

  • array-based: stores data in an array
  • reference-based: stores data in separate “nodes”, each of which has a reference to another node in the data structure

Generics

  • defining data structures without generics
    • option 1: define a class for each type we want to work with
      • lots of duplicate code if we want to use the data structure for many different types
      • compiler will still help us with type checking, preventing many bugs
    • option 2: define one class that takes Objects
      • any type can be stored as an Object, so we can put whatever we want in our data structure
      • compiler can’t type check, which allows for more bugs
      • data structure can be heterogeneous (mix of different types)
  • generic class: add type parameter(s)
    • we specify the type parameter in code using the class, which lets us declare instances with different type parameters
    • compiler will handle type checking, preventing bugs
    • we only have one class definition that works with any type
    • we can still make it heterogeneous by specifying Object as the type parameter
  • java generics can be tricky
    • good tutorial: https://docs.oracle.com/javase/tutorial/java/generics/index.html
    • type erasure: types are used at compile time for type checking, but everything is an Object at runtime
    • can’t be used with primitive types, so we must use boxed types (Integer, Double, Character, etc.) instead
    • can’t instantiate a generic array (new E[100]), but we can still declare a generic array type and cast Object[] to a generic array type ((E[]) new Object[100])
    • similar languages, like C#, may have slightly different syntax for generics
